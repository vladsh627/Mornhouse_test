# Mornhouse - Product Analytics Test Task

## Завдання 1: Розрахунок сукупного доходу сегменту Trial

### 1. Очищення даних та виявлення інсайтів
Під час аналізу вхідного датасету було виявлено наступні особливості:
* **Структура**: Довідкова інформація (метадані) була об'єднана з транзакційними даними. Для аналізу залишено лише перші 4 колонки.
* **Логіка прапорця Trial**: Згідно з документацією, `is_trial = 0` означає підписку **з** тріалом (інверсована логіка).
* **Ціноутворення Trial**: Тріал є платним ($6.99), подальші ребіли коштують $29.99.

### 2. Припущення (Assumptions)
Для розрахунку метрики "Сукупний дохід" (Cumulative Revenue) були прийняті наступні продуктові припущення:
1. **Gross Revenue**: Розраховується "брудний" дохід (до вирахування комісій App Store / Google Play 15-30% та податків).
2. **Success Rate**: Вважаємо, що всі транзакції (First payment та заявлена кількість ребілів) були успішно списані.
3. **Refund Rate = 0%**: Відсутність чарджбеків та повернень.

### 3. Формула розрахунку
Для кожного користувача в сегменті `is_trial = 0` дохід рахувався за формулою:
`User Revenue = 6.99 + (subscription_renewal_amount * 29.99)`
Для кожного користувача в сегменті `is_trial = 1` дохід рахувався за формулою:
`User Revenue = 40 + (subscription_renewal_amount * 40)`

Сукупний дохід — це сума `User Revenue` всіх користувачів даного сегменту.

### 4. Інструкція із запуску
Для відтворення процесу очищення даних та розрахунку:
1. Встановіть залежності: `pip install -r requirements.txt`
2. Запустіть скрипт: `python scripts/data_preparation.py`
3. Очищений датасет буде збережено у папку `data` для подальшої візуалізації в Tableau.

**Результат Завдання 1:** Сукупний дохід склав $1,745,581.66.

### Структура дашборду:
*   **ARPU: Trial vs No Trial:** Додаткова візуалізація, що відповідає на головне бізнес-питання — який сегмент генерує більший середній дохід на одного користувача (Average Revenue Per User).

## 🚩 Exploratory Data Analysis (EDA) & Data Quality Insights
Під час аналізу сирих даних перед побудовою візуалізацій було виявлено низку аномалій. Ці знахідки можуть свідчити про проблеми з логуванням подій або баги в системі білінгу:

1. **Аномалія форматів дат (Date Format Ambiguity):** 
   У датасеті присутні лише 8 унікальних дат: 7 значень з `7/1/19` по `7/7/19` та одне `1/1/22`. Існує висока ймовірність помилки форматування даних (`MM/DD/YY` замість `DD/MM/YY`).
   Через цю дискретність і розрив у часі використання стандартних безперервних часових фільтрів на дашборді замінено на точковий Dropdown-фільтр.

3. **Дублікати та "Billing Bug" (Multiple same-day rebills):**
   У датасеті 9 426 рядків, проте лише 3 188 унікальних `user_id`. Деякі користувачі зустрічаються в таблиці багато разів, маючи по 5-10 ребілів *в один і той самий день*. Це виглядає як технічний баг платіжного шлюзу або відсутність агрегації на рівні SQL-запиту при вивантаженні таблиці. 

4. **Критичний Churn у сегменті "No Trial":**
   Для користувачів без тріалу (`is_trial = 1`) виявлено фактичну відсутність Retention. З усієї когорти лише *один* клієнт здійснив репеймент. Подальший аналіз цього сегменту варто будувати виключно на порівнянні їхнього стартового платежу ($40) з LTV когорти тріалу, оскільки довгострокове утримання таких користувачів дорівнює нулю. А бо ж що ймовірно, така анамолія пов'язана, як і минулі, з технічною стороною, на етапі збору інформації.
